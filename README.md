# Atrucks → Google Sheets → ATI.SU

Сервис каждые `POLL_INTERVAL_MINUTES` минут забирает список аукционов
(«грузов») с Atrucks (`/carrier/auctions/lots/general/quick/`) и
записывает их в Google-таблицу. **Публикация на ATI.SU теперь ручная** —
кнопка прямо в таблице (Apps Script), а не автоматическая.

Исключение: если лот пропадает с Atrucks, а в таблице у него уже стоит
`ATI_cargo_id` (то есть он был опубликован) — карточка на ATI снимается
автоматически (`DELETE /v2/cargos/{id}`). Это уборка уже принятого ранее
решения, а не новая публикация.

## Архитектура

```
Atrucks (аукционы)
   │  каждые POLL_INTERVAL_MINUTES минут
   ▼
Node-сервис (Railway) — src/sync.js
   │  пишет/обновляет строки, удаляет пропавшие лоты
   ▼
Google-таблица
   ├── лист "Логисты"     — справочник Клиент -> Логист/Токен/ContactId,
   │                         редактируете сами, без участия кода
   └── лист "Лоты Atrucks" — лоты + чекбокс "Отправить"
          │  кнопка "Опубликовать отмеченные" (Apps Script,
          │  AtrucksSheetPublish.gs)
          ▼
        ATI.SU (POST/PUT /v2/cargos)
```

Привязка "клиент → логист" больше не зашита в коде (`logists.js`
оставлен только как референс для первоначального переноса) — она живёт
в листе "Логисты" и подтягивается по точному совпадению названия
клиента.

## Логика цены

Для каждого лота берётся `start_price` (ставка с НДС на Atrucks):

```
rate          = round((start_price / 1.22) * 0.85)
rate_with_vat = round(rate * 1.22)
```

Ставка перевозчика на ATI — на 15% ниже ставки клиента без НДС
(`config.pricing.factor = 0.85`). Эта цена пересчитывается и
обновляется в таблице каждый цикл синхронизации (каждые
`POLL_INTERVAL_MINUTES` минут) независимо от того, опубликован лот на
ATI или ещё нет. Сама публикация/обновление на ATI происходит только
по нажатию кнопки в таблице.

## Откуда берётся название клиента

Список лотов Atrucks отдаёт только числовой `company_id`, без названия
компании. Настоящее название можно получить лишь медленным запросом
HTML-страницы каждого лота (`src/list-company-names.js`, ловит 429 при
частых запросах) — не годится для регулярного цикла на 1000+ лотов.

Поэтому используется статический словарь `src/companyNames.js`
(`company_id -> название`), уже содержащий ~30 известных клиентов.
Для нового, ещё не опознанного `company_id` в таблице появится
плейсхолдер `Неизвестно (id=XXXX)` — заметно и не блокирует попадание
лота в таблицу. Опознать вручную (посмотреть лот в Atrucks или разово
запустить `list-company-names.js`) и дописать строку в словарь.

## Логика типа кузова

`transport:truck_kinds` (строка вида "Тент, Изотерм, Рефрижератор")
парсится по ключевым словам в `src/mapper.js` (`BODY_TYPE_RULES`).
Если ничего не распознано — используется дефолтный тип (тент, код `200`,
см. `config.ati.defaultBodyType`).

**Важно:** коды `body_types` (201, 202, 203...) для остальных типов кузова
(реф, борт, контейнеровоз и т.д.) в коде указаны по best-guess и нуждаются
в проверке через реальную публикацию или официальный справочник ATI.SU.

## Шаг 1. Создание Google-таблицы

1. Создайте новую Google-таблицу.
2. Переименуйте первый лист в `Логисты`, во второй строке и далее
   заполните колонки: `A` Клиент (точно как в `companyNames.js`),
   `B` ФИО логиста, `C` Токен ATI, `D` Contact ID ATI.
3. Создайте второй лист `Лоты Atrucks`. В первой строке — заголовки
   (любые, для себя): `ext_id, Клиент, Логист, Откуда, Куда, Груз, Вес,
   Объём, Тип кузова, Ставка без НДС, Ставка с НДС, Дата погрузки,
   Дата выгрузки, Статус, ATI_cargo_id, Отправить, ATI_Body_JSON,
   Обновлено` (колонки A..R по порядку).
4. Расширения → Apps Script → вставьте содержимое `AtrucksSheetPublish.gs`
   из этого репозитория → Сохранить.
5. В Apps Script запустите функцию `setupSheet` один раз вручную
   (выделите её в выпадающем списке функций и нажмите ▶) — она поставит
   чекбоксы в колонку "Отправить" и формулу подбора логиста в колонку
   "Логист". Согласитесь на запрос разрешений.
6. Обновите страницу таблицы — в меню появится "🚛 Atrucks → ATI".

## Шаг 2. Сервис-аккаунт Google (для записи из Node)

1. В [Google Cloud Console](https://console.cloud.google.com/) создайте
   проект (или используйте существующий).
2. APIs & Services → Library → найдите **Google Sheets API** → Enable.
3. APIs & Services → Credentials → Create Credentials → Service account.
   Имя любое, роль можно не назначать (права даются прямо в таблице).
4. Откройте созданный сервис-аккаунт → Keys → Add Key → Create new key →
   JSON. Скачается файл — он понадобится один раз, дальше не нужен.
5. Откройте вашу Google-таблицу → Поделиться → вставьте e-mail
   сервис-аккаунта (поле `client_email` из JSON) → права **Редактор**.
6. Из скачанного JSON возьмите:
   - `client_email` → переменная `GOOGLE_SERVICE_ACCOUNT_EMAIL`
   - `private_key` → переменная `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY`
     (это многострочная строка с `-----BEGIN PRIVATE KEY-----...`;
     на Railway вставляется как есть — переносы строк там можно
     указывать как буквальные `\n`, код их сам разворачивает обратно)
   - ID таблицы из её URL (`.../spreadsheets/d/ЭТОТ_ID/edit`) →
     переменная `GOOGLE_SHEET_ID`

## Шаг 3. Получение ATRUCKS_COOKIE

1. Зайдите в личный кабинет atrucks.su под своим логином.
2. Откройте DevTools → Network → XHR.
3. Откройте раздел "Аукционы всех контрагентов" (Аукционы → Аукционы всех
   контрагентов), убедитесь, что появляется запрос `quick/?page=...`.
4. Кликните на запрос → Headers → Request Headers → найдите заголовок
   `Cookie`.
5. Скопируйте **всё значение целиком** (одна длинная строка вида
   `__ddg1_=...; sessionid=...; csrftoken=...; ...`) в переменную
   `ATRUCKS_COOKIE`.

⚠️ Кука `sessionid` со временем истекает. Когда сервис начнёт получать
ошибки "истекла сессия/cookie" (видно в логах Railway), нужно повторить
эти шаги и обновить `ATRUCKS_COOKIE` в переменных окружения Railway.

## Шаг 4. Проверка маппинга (dry-run)

```bash
cp .env.example .env
# заполните ATRUCKS_COOKIE в .env
node src/test-dry-run.js
```

Скрипт прогонит все лоты через маппинг (без публикации и без записи в
Sheets), покажет распределение по клиентам и неопознанные `company_id`.

## Шаг 5. Развёртывание на Railway

1. Создайте новый проект на Railway, подключите этот репозиторий
   (Dockerfile уже включён).
2. Добавьте **volume**, смонтированный в `/data` (для SQLite).
3. Задайте переменные окружения (см. `.env.example`):
   - `ATRUCKS_COOKIE`
   - `GOOGLE_SHEET_ID`, `GOOGLE_SERVICE_ACCOUNT_EMAIL`,
     `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY`
   - `ATI_BOARD_ID` (нужен для тела запроса всем лотам)
   - `POLL_INTERVAL_MINUTES` (рекомендуется 10)
   - `DB_PATH=/data/atrucks-ati.sqlite`
4. Деплой. Сервис сразу выполнит первый цикл синхронизации и далее —
   по расписанию.

## Эндпоинты

- `GET /health` — статус последнего запуска и краткая статистика
- `POST /run` — запустить цикл синхронизации вручную (асинхронно)

## Структура проекта

```
src/
  config.js          — конфигурация и константы (включая Google Sheets)
  companyNames.js     — словарь company_id -> название клиента
  db.js              — SQLite (node:sqlite): кэш "не менялось" и city_cache
  atrucksClient.js    — получение лотов с Atrucks (пагинация)
  atiCities.js        — резолвинг city_id через ATI autocomplete (с кэшем)
  sheetsClient.js      — чтение/запись Google Sheets (лоты + логисты)
  mapper.js           — маппинг лота Atrucks -> тело ATI /v2/cargos
  atiClient.js         — POST/PUT/DELETE /v2/cargos (вызывается Node только
                          при автоматическом снятии пропавших лотов)
  logists.js           — УСТАРЕЛО, оставлен как референс
  sync.js              — основной цикл: Atrucks -> Google Sheets
  index.js             — точка входа, расписание, healthcheck
  test-dry-run.js      — тестовая проверка маппинга без публикации

AtrucksSheetPublish.gs — Apps Script для самой Google-таблицы:
                          кнопка ручной публикации/обновления на ATI
```
