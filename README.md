# Atrucks → ATI.SU bridge

Сервис каждые `POLL_INTERVAL_MINUTES` минут забирает список аукционов
(«грузов») с Atrucks (`/carrier/auctions/lots/general/quick/`) и публикует
их на ATI.SU как свои грузы (`POST/PUT /v2/cargos`), от имени firm_id из
`.env`. Лоты, исчезнувшие с Atrucks, снимаются с публикации на ATI
(`DELETE /v2/cargos/{id}`).

## Логика цены

Для каждого лота берётся `start_price` (ставка с НДС на Atrucks):

```
rate_with_vat = round(start_price * 0.85)
rate          = round(rate_with_vat / 1.22)
```

То есть на ATI публикуется ставка на 15% ниже, чем на Atrucks.

## Логика типа кузова

`transport:truck_kinds` (строка вида "Тент, Изотерм, Рефрижератор")
парсится по ключевым словам в `src/mapper.js` (`BODY_TYPE_RULES`).
Если ничего не распознано — используется дефолтный тип (тент, код `200`,
см. `config.ati.defaultBodyType`).

**Важно:** коды `body_types` (201, 202, 203...) для остальных типов кузова
(реф, борт, контейнеровоз и т.д.) в коде указаны по best-guess и нуждаются
в проверке через реальную публикацию или официальный справочник ATI.SU.
Перед полным запуском стоит свериться со справочником типов ATI и при
необходимости подправить таблицу `BODY_TYPE_RULES`.

## Шаг 1. Получение ATRUCKS_COOKIE

1. Зайдите в личный кабинет atrucks.su под своим логином.
2. Откройте DevTools → Network → XHR.
3. Откройте раздел "Аукционы всех контрагентов" (Аукционы → Аукционы всех
   контрагентов), убедитесь, что появляется запрос `quick/?page=...`.
4. Кликните на запрос → Headers → Request Headers → найдите заголовок
   `Cookie`.
5. Скопируйте **всё значение целиком** (одна длинная строка вида
   `__ddg1_=...; sessionid=...; csrftoken=...; ...`) в переменную
   `ATRUCKS_COOKIE`.

⚠️ Кука `sessionid` со временем истекает (логаут по таймауту или явный
выход из аккаунта). Когда сервис начнёт получать ошибки
"истекла сессия/cookie" (видно в логах Railway), нужно повторить эти шаги
и обновить `ATRUCKS_COOKIE` в переменных окружения Railway, после чего
сервис перезапустится автоматически.

## Шаг 2. Проверка маппинга (dry-run)

Перед полным запуском рекомендуется локально проверить, что данные
парсятся корректно:

```bash
cp .env.example .env
# заполните ATRUCKS_COOKIE и при необходимости ATI_* в .env
node src/test-dry-run.js
```

Скрипт выведет первые 3 лота с первой страницы и результат их маппинга
в тело ATI (включая определённые `city_id`, `body_types`, цену, даты).
Проверьте корректность city_id (через ручной поиск на ati.su) и
body_types перед запуском полной автоматизации.

## Шаг 3. Развёртывание на Railway

1. Создайте новый проект на Railway, подключите этот репозиторий
   (Dockerfile уже включён).
2. Добавьте **volume**, смонтированный в `/data` (для SQLite).
3. Задайте переменные окружения (см. `.env.example`):
   - `ATRUCKS_COOKIE`
   - `ATI_TOKEN`, `ATI_FIRM_ID`, `ATI_CONTACT_ID`, `ATI_BOARD_ID`
   - `POLL_INTERVAL_MINUTES` (рекомендуется начать с 10-15)
   - `DB_PATH=/data/atrucks-ati.sqlite`
4. Деплой. Сервис сразу выполнит первый цикл синхронизации и далее —
   по расписанию.

## Эндпоинты

- `GET /health` — статус последнего запуска и краткая статистика
- `POST /run` — запустить цикл синхронизации вручную (асинхронно)

## Первый запуск с большим объёмом (1000+ грузов)

При первом запуске сервис попытается опубликовать **все** текущие лоты
с Atrucks (порядка 1000+). Рекомендации:

- Начните с `POLL_INTERVAL_MINUTES` побольше (например, 30-60) на первый
  день, чтобы было проще отслеживать логи и ошибки.
- Следите за логами Railway на предмет ошибок ATI (лимиты на количество
  активных грузов, неверные `city_id`/`body_types` и т.п.) и
  корректируйте `src/mapper.js` по результатам.
- Если ATI вернёт ошибку лимита — потребуется либо обсудить с ATI
  увеличение лимита, либо ввести фильтрацию лотов (например, по
  `company_id`, направлению, цене) — структура кода это позволяет
  добавить позже в `src/sync.js`.

## Структура проекта

```
src/
  config.js        — конфигурация и константы
  db.js            — SQLite (node:sqlite), таблицы mapping и city_cache
  atrucksClient.js — получение лотов с Atrucks (пагинация)
  atiCities.js     — резолвинг city_id через ATI autocomplete (с кэшем)
  mapper.js        — маппинг лота Atrucks -> тело ATI /v2/cargos
  atiClient.js     — POST/PUT/DELETE /v2/cargos
  sync.js          — основной цикл синхронизации
  index.js         — точка входа, расписание, healthcheck
  test-dry-run.js  — тестовая проверка маппинга без публикации
```
